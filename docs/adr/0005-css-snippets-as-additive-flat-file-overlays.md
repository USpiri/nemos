# CSS Snippets as additive flat-file overlays

CSS Snippets are flat `.css` files, not folders. Themes use a folder-based structure (`themes/[ID]/theme.css`) because a theme is a named, selectable artifact that may grow to include metadata alongside its CSS. Snippets are not selectable — all of them are active (modulo per-workspace toggles) — and there is nothing to put beside a single CSS file, so the folder adds friction with no payoff. A flat file is also easier to drop in via a file manager and matches the mental model users have from Obsidian.

Snippets across the two scopes (Global and Workspace) are loaded additively. The Theme system resolves ID collisions by letting the Workspace scope shadow the Global scope. We rejected that model for Snippets because the collision assumption breaks down: a user might deliberately maintain a global `styles.css` and a workspace `styles.css` as two complementary layers, not as overrides of each other. Additive loading makes this natural. Any conflict between the two files is resolved by CSS cascade — Global Snippets are injected first, Workspace Snippets after, so Workspace rules win on conflict without any application-level logic.

Toggle state is per-Workspace, not per-Snippet-scope. "Global" describes a Snippet's *availability* (reachable from all Workspaces), not its *enforcement* (active in all Workspaces regardless of context). A user may want a Global Snippet active in a coding workspace but suppressed in a journaling workspace. Per-Workspace toggle state enables this; global toggle state does not. Only disabled Snippet IDs are persisted (under `disabledGlobalSnippets` and `disabledWorkspaceSnippets` in the Workspace Settings delta), so a newly dropped-in Snippet is active immediately without any settings update.
